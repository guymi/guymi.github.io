In a perfect world, our infrastructure stays connected, our clusters are healthy, and our certificates never expire. In the real world, security means rotation — and rotation often means broken connections.

Recently, I tackled a challenge involving **ArgoCD connectivity** that was costing our Operations team hours of manual work and unnecessary downtime. Here is how I used the **External Secrets Operator (ESO)** to automate the entire lifecycle of cluster credentials.

The Pain: The 90-Day Disconnect
-------------------------------

Our organization uses ArgoCD to manage workloads across several GKE clusters. These clusters host sub-clusters provisioned by a dedicated operator. For security reasons, the certificates for these sub-clusters (`ca.crt`, `tls.crt`, `tls.key`) rotate every 90 days.

**The old workflow looked like this:**

1.  Certificates rotate.
2.  ArgoCD loses connectivity to the sub-cluster.
3.  An Operations engineer realizes there’s an outage.
4.  The engineer manually extracts new credentials via `kubeconfig`.
5.  They run `argocd cluster add...` for up to 12 different clusters.

![The 90-Days problem](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*O64PQiF6YcwgARrXwIU6mQ.png)

The Strategy: Why Build When You Can Reuse?
-------------------------------------------

I needed a solution that was **automated and simple.** While I could have written a custom script or a `CronJob`, I realized we already had the perfect tool in our stack: the **External Secrets Operator (ESO)**. Most people use ESO to pull secrets _from_ a vault _into_ Kubernetes. However, ESO has a lesser-known capability: the `PushSecret` resource.

By combining `PushSecret` and `ExternalSecret`, I could create a "Secret Relay":

*   **Step 1:**
    Push rotated credentials from the hosting cluster to GCP Secret Manager.
*   **Step 2:**
    Pull those credentials into the ArgoCD cluster and update the existing ArgoCD cluster secret.

This reuses the operator’s internal reconciliation loop, meaning no custom code to maintain.

The Implementation
------------------

![from certificate rotation to ArgoCD secret](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*z0c7Sem2XEnmv0oRd9RE0A.png)

1. Pushing from the Control Cluster
------------------------------------

First, we need to get the “source of truth” into a vault. We grant the ESO Google Service Account `secretmanager.admin` permissions and apply a `PushSecret` manifest.

```
# A snippet of the PushSecret configuration
apiVersion: external-secrets.io/v1alpha1
kind: PushSecret
metadata:
  name: pushsecret-cluster-admin-secret
  namespace: management
spec:
  # The provider's secret will be deleted if the PushSecret is deleted
  deletionPolicy: Delete
  # How often the secret will be pushed
  refreshInterval: 1h
  # A list of secret stores to push secrets to
  secretStoreRefs:
    - name: cluster-secret-store
      kind: ClusterSecretStore
  # The source Kubernetes secret to be pushed
  selector:
    secret:
      name: cluster-client-admin
  # Defines which keys from the source secret to push
  data:
    - match:
        # The key in the source Kubernetes secret
        secretKey: ca.crt
        # The name of the secret in GCP Secret Manager
        remoteRef:
          remoteKey: prod1-cluster-admin-secret-ca-crt
    - match:
        secretKey: tls.crt
        remoteRef:
          remoteKey: prod1-cluster-admin-secret-tls-crt
    - match:
        secretKey: tls.key
        remoteRef:
          remoteKey: prod1-cluster-admin-secret-tls-key
```

_This ensures that every hour, GCP Secret Manager is updated with the latest certs from the cluster._

2. Pulling into the ArgoCD Cluster
-----------------------------------

Next, on the cluster where ArgoCD lives, we use an `ExternalSecret`. The trick here is using a **template** to format the secret exactly how ArgoCD expects it—wrapped in a JSON `config` block.

```
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: cluster-mycluster.com-******
  namespace: argocd
spec:
  # How often to check for changes in the source secret
  refreshInterval: 1h
  # Reference to the SecretStore that knows how to connect to GCP
  secretStoreRef:
    name: cluster-secret-store
    kind: ClusterSecretStore
  # Defines the target Kubernetes secret to be created
  target:
    # The name of the Kubernetes secret that will be created/updated
    name: cluster-mycluster.com-******
    # Update existing secret instead of creating it
    creationPolicy: Merge
    # Retain ArgoCD's secret if the ExternalSecret is deleted
    deletionPolicy: Retain
    # Template for the created secret's data
    template:
      data:
        config: |
          {
            "tlsClientConfig": {
              "insecure": false,
              "caData": "{{ .caData | b64enc }}",
              "certData": "{{ .certData | b64enc }}",
              "keyData": "{{ .keyData | b64enc }}"
            }
          }
  # Defines which secrets to fetch from the provider
  data:
    # Key name in the target K8s secret
    - secretKey: caData
      remoteRef:
        # Secret name in GCP
        key: prod1-cluster-admin-secret-ca-crt
    - secretKey: certData
      remoteRef:
        key: prod1-cluster-admin-secret-tls-crt
    - secretKey: keyData
      remoteRef:
        key: prod1-cluster-admin-secret-tls-key
```

_By setting_ `_creationPolicy: Merge_`_, we update the existing ArgoCD secret without overwriting other metadata required by ArgoCD.
By setting_ `_deletionPolicy: Retain_`_, we make sure ArgoCD secret remains intact if this_ `_ExternalSecret_` _resource is deleted._

The Result: Set It and Forget It
--------------------------------

By leveraging the reconciliation loops of ESO, we’ve reduced a multi-step manual process into a self-healing system.

*   **Zero Manual Intervention:**
    Operations engineers no longer need to run CLI commands every 90 days.
*   **Reduced Downtime:**
    The gap between rotation and restoration is now, at most, one hour (and can be narrowed further by adjusting `refreshInterval`).
*   **Cleaner Architecture:**
    No custom scripts, no extra overhead — just native Kubernetes resources.

![Using ESO to also push secrets…](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*8UyYA7kA-GYqP7fFkNIsjQ.png)

**Key Lesson:** Sometimes the best tool for the job is the one you’re already using. Before you write a new script, look at the operators already running in your cluster; they might have a “hidden” feature that solves your problem.
