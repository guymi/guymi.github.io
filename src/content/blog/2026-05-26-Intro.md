_How we stopped copying gigabytes of model files between servers and got 6.9 GB/s warm reads instead_

> **_AWS S3 Files launched on April 7, 2026._** _This post is one of the first real-world accounts of using it in production._

The Problem We Were Trying to Solve
-----------------------------------

We run a platform that serves ML inference at scale. Our architecture involves a fleet of EC2 instances — algo machines — each running multiple services that load model files on every request. For a long time, our model storage strategy was embarrassingly manual: copy the files from one EC2 to another whenever there was a new version.

It worked. Until it didn’t.

As our model sizes grew and our instance count increased, the cracks started showing:

*   New instances took minutes to become ready, waiting for multi-gigabyte copies to complete
*   Version promotions were risky — we had no clean way to roll back
*   We discovered a race condition: if a service was reading model files while a copy was in progress, it would silently load a mix of old and new weights. The kind of bug that makes your inference results subtly wrong and takes hours to diagnose
*   There was no separation between prod and dev model versions

We knew we needed a better solution. What we didn’t expect was how far the journey would take us.

Attempt 1: mountpoint-s3
------------------------

Our first instinct was **mountpoint-s3** — AWS’s open-source tool that lets you mount an S3 bucket as a local filesystem using FUSE. The appeal was obvious: model files live in S3, instances mount the bucket, no more copying.

We designed a versioning scheme using S3 prefixes:

```
s3://my-company-models-prod/
  models-v1.0.0/
  models-v2.0.0/
  models-v3.0.0/   ← active version
```

Each instance mounts with `--prefix models-v3.0.0/` so services always read from `/mnt/models/` without knowing which version they're on. Version promotion becomes a prefix switch — no file copies, no race conditions.

Clean in theory. Then we ran the benchmarks.

The FUSE ceiling
----------------

On our r5.4xlarge instances (128 GiB RAM), we measured:

Read type Speed Cold (from S3) ~327 MB/s Warm (from cache) ~1.5 GB/s Direct EBS read ~4.7 GB/s

The warm read through mountpoint-s3 was capped at **1.5 GB/s** — significantly slower than a direct EBS read of the same cached file at 4.7 GB/s. The culprit was FUSE overhead: every read, even from RAM, goes through the mountpoint-s3 process consuming 60–70% of a single CPU core. FUSE is single-threaded per file, and it was the bottleneck regardless of where the data lived.

We could live with 1.5 GB/s. But there were more problems:

*   Cache is **per-instance** and stored on EBS — clearing it on every version promotion added complexity
*   Version promotion required SSM commands to every instance: stop mount → clear cache → update prefix → restart mount — a ~5–10 second gap per instance during which requests would fail
*   The EBS throughput on our t3.xlarge instances was actually **slower** than cold S3 reads, making the cache counterproductive on smaller instances

We had solved the race condition and the manual copying. But we had introduced operational complexity. Then AWS announced something new.

Discovering AWS S3 Files
------------------------

[**AWS S3 Files**](https://aws.amazon.com/s3/features/files/) (launched on April 7, 2026) is a managed shared file system that connects EC2 instances directly to S3 data using NFS 4.2, built on the EFS infrastructure. S3 Files works across the full AWS compute stack — EC2, Lambda, ECS, EKS, Fargate, and Batch. It’s fundamentally different from mountpoint-s3:

![captionless image](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*VTR83w70LBmNmoRgR2EV0A.png)

mountpoint-s3 S3 Files Protocol FUSE (userspace) NFS 4.2 (kernel) Cache Per-instance EBS Per-instance high-performance storage File system semantics Limited Full POSIX (symlinks, locking, permissions) Write support Read-only Read/write Mount helper Custom binary amazon-efs-utils

The NFS approach means reads go through the kernel’s page cache directly — no FUSE overhead, no userspace bottleneck.

We set it up and ran the same benchmarks.

The numbers
-----------

On r5.4xlarge with a 181 MB model file:

Read type mountpoint-s3 S3 Files Cold (first read) ~327 MB/s ~127 MB/s Warm (subsequent) ~1.5 GB/s **~6.9 GB/s**

Cold reads are slower with S3 Files — NFS has more overhead than mountpoint-s3’s direct S3 streaming. But warm reads are **4.5x faster** because NFS goes straight to the OS page cache in RAM, bypassing any middleware layer.

For our use case — small model files (50–180 MB) read on every inference request — the warm read is what matters. After the first request, every subsequent read completes in **~26 milliseconds**. Effectively instantaneous.

The Symlink Insight
-------------------

S3 Files supports full POSIX file system semantics, including **symlinks that persist back to the S3 bucket**. This turned out to be the key to an elegant version promotion strategy.

Our bucket structure:

```
s3://my-company-models-prod/
  models-v1.0.0/      ← old version (retained for rollback)
  models-v2.0.0/      ← old version (retained for rollback)
  models-v3.0.0/      ← current version
  latest -> models-v3.0.0/   ← symlink
```

Services always read from `/mnt/models/latest`. The path never changes. Version promotion is a single command run from any one instance:

```
# Create temp read-write mount (instances are read-only)
sudo mount -t s3files -o rw prod-file-system-id:/ /mnt/models-rw
sudo ln -sfn models-v3.0.0 /mnt/models-rw/latest   # relative symlink!
sudo umount /mnt/models-rw
```

![captionless image](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*qZrKnfj43jzvjI8hvKsgMQ.png)

That’s it. All instances immediately see the new version through their read-only mounts — no SSM commands to each instance, no mount restarts, no cache clearing.

Symlinks in the filesystem vs the AWS Console
---------------------------------------------

One practical gotcha worth knowing: the symlink behaves exactly as expected when accessed through the mounted filesystem. On any instance you can see it clearly:

```
ls -la /mnt/models/
# latest -> models-v3.0.0
```

However, **in the AWS S3 Console the symlink appears as a regular file**. There is no visual indication that `latest` is a symlink or which version it points to. If you look at the bucket contents in the console you will just see a file called `latest` with no destination shown.

This means you should not rely on the console to check which version is currently active. Instead, check it from any mounted instance or store the current version in a separate plain text file alongside the symlink as a human-readable reference for the console

```
echo "models-v3.0.0" | sudo tee /mnt/models-rw/current-version.txt
```

Why relative symlinks matter
----------------------------

An important detail: the symlink must use a **relative path**, not an absolute one.

```
# WRONG — absolute path
sudo ln -sfn /mnt/models-rw/models-v3.0.0 /mnt/models-rw/latest
# Resolves as: /mnt/models/latest → /mnt/models-rw/models-v3.0.0 ← broken on other instances
```

```
# CORRECT — relative path
sudo ln -sfn models-v3.0.0 /mnt/models-rw/latest
# Resolves as: /mnt/models/latest → /mnt/models/models-v3.0.0 ← works everywhere
```

No race condition
-----------------

One concern with symlink-based promotion is whether the cache could serve stale data. We tested it: **S3 Files cache is object-based**, not path-based. When the symlink points to a new version folder, the objects under that folder have never been cached — so the first read is always a cold read from S3, and there’s no possibility of mixing old and new model files.

Architecture: Two Buckets, Two Environments
-------------------------------------------

We maintain strict separation between production and development:

```
my-company-models-prod    →  prod/staging instances  (read-only mount)
my-company-models-dev     →  dev/rungen/ephemeral     (read-only mount)
```

Each bucket has its own S3 Files file system. Since our prod and dev instances are in different VPCs, each VPC needs its own **mount target** — but they can share the same file system and data.

One important prerequisite: S3 Files requires **S3 versioning to be enabled** on the bucket. For our read-only model files this adds negligible cost since we never overwrite objects — each model version lives in its own folder.

The Migration Script
--------------------

We wrote a single bash script that handles the full migration for any instance:

```
./run_migration.sh <BUCKET_NAME> <FILE_SYSTEM_ID> <INITIAL_VERSION>
```

```
# Example:
./run_migration.sh my-company-models-dev fs-002a95988843125a6 models-v0.0.1
```

The script:

1.  Installs `amazon-efs-utils` (v3.0.0+ required for S3 Files support)
2.  Mounts the file system read-only at `/mnt/models`
3.  Adds an fstab entry so the mount survives reboots (`_netdev,nofail,ro`)
4.  Updates `docker-compose.yaml` to replace old EFS paths with `/mnt/models/latest`
5.  Restarts only the services that consume model files
6.  Runs a smoke test (mount active + health check on port 5060)

We rolled it out in three phases via AWS SSM Run Command — no SSH required:

*   **Phase 1**: Dedicated throwaway instance (`model-migration-EC2`) to validate end-to-end
*   **Phase 2**: All dev bucket instances in parallel
*   **Phase 3**: All prod bucket instances

Lessons Learned
---------------

**1. Benchmark on your actual instance type**

On a t3.xlarge, mountpoint-s3 cache was actually slower than cold S3 reads due to EBS throughput limitations. On r5.4xlarge, S3 Files warm reads hit 6.9 GB/s. Hardware matters enormously for storage performance.

**2. FUSE overhead is real and significant**

mountpoint-s3 consumed 60–70% of a single CPU core during reads, capping throughput at ~1.5 GB/s regardless of whether data was in RAM or on disk. NFS avoids this entirely by operating in kernel space.

**3. Relative symlinks, not absolute**

A small detail that matters a lot when the same filesystem is mounted at different paths by different processes (e.g. a read-write CI/CD mount vs read-only instance mounts).

**4. Object-based caching eliminates race conditions**

Unlike a `latest/` folder copy (which is not atomic) or a `version.txt` pointer (which has a window where the pointer is updated before the new files are ready), symlink-based promotion with object-based caching is inherently race-condition free.

**5. S3 Files versioning cost is negligible for read-only workloads**

S3 object versioning is required by S3 Files. For model files that are never overwritten — each version lives in its own prefix folder — this adds essentially zero cost. You only pay for multiple versions of the same object key, which never happens in our setup.

What’s Next
-----------

We’re planning to migrate from EC2 to EKS. The good news: S3 Files mounts in Kubernetes using the **EFS CSI Driver** via a `PersistentVolume` with `ReadOnlyMany` access mode. The symlink-based version promotion stays exactly the same — no application changes required.

Summary
-------

Before After Model delivery Manual EC2-to-EC2 copy S3 Files NFS mount Version promotion Copy files (race condition risk) Update symlink (atomic) Warm read speed N/A (always copying) ~6.9 GB/s New instance ready Minutes (waiting for copy) Seconds (mount and go) Rollback Re-copy old files Update symlink back Prod/dev isolation None Separate buckets + VPCs

AWS S3 Files is still new and not widely discussed yet, but for teams running ML workloads on EC2 with model files stored in S3, it’s worth a serious look. The combination of NFS semantics, object-based caching, and symlink support makes it significantly more capable than mountpoint-s3 for our use case — and the operational simplicity of the version promotion flow is something we didn’t expect to achieve this cleanly.
